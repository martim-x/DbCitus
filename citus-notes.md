## Введение

_**Citus**_ — это расширение PostgreSQL, которое добавляет шардинг и распределённые запросы по нескольким нодам. На твоём стенде один координатор и три воркера объединены в единый кластер, доступный как обычный Postgres через координатор.

#citus
## Что такое Citus и зачем он нужен

_**Citus**_ превращает кластер из нескольких экземпляров PostgreSQL в одну логическую базу с горизонтальным масштабированием. Он нужен, когда одного сервера PostgreSQL не хватает по объёму данных или нагрузке, но хочется остаться на SQL и протоколе Postgres. 

Основные задачи:

- распределить данные таблиц по нескольким серверам (шардинг);
- выполнять запросы параллельно на воркерах и агрегировать результат; 
- прозрачно для клиента — приложение видит один `postgres://...`, а не десяток.

## Архитектура на твоём примере

### Роли нод

- _**Координатор**_ (`citus_coordinator`):
  - принимает все подключения (pgAdmin, `psql`, приложения) на `localhost:55432`;
  - хранит метаданные Citus в `pg_dist_*`, но не держит данные распределённых таблиц; 
  - планирует и оркестрирует распределённые запросы. 

- _**Воркеры**_ (`citus_worker_1/2/3`):
  - обычные Postgres 17 с установленным Citus;  
  - хранят физические таблицы‑шарды (`dist_test_1020xx` и т.п.); 
  - выполняют под‑запросы, которые им прислал координатор.

- _**Менеджер**_ (`citus_manager`):
  - подключается к координатору (`CITUS_HOST=citus-coordinator`); 
  - отслеживает docker‑контейнеры с меткой воркера и автоматически регистрирует/удаляет их в кластере (то, что ты видел как `adding citus_worker_1` и т.п.). 

### Метаданные кластера

На координаторе состояние кластера описывают:

- `pg_dist_node` — список нод (координатор и воркеры) с `groupid`, именем и портом; 
- `pg_dist_shard` — список шардов распределённых таблиц с `shardid` и диапазоном хэша (`shardminvalue`, `shardmaxvalue`); 
- `pg_dist_placement` — «куда положен шард»: на какую ноду (`groupid`), его состояние и размер; 
- `citus_shards` — удобное представление: логическая таблица → её шарды → физические имена таблиц. 

На твоём выводе `pg_dist_shard` видно, что:

- у `dist_test` создано 32 шарда (`shardid 102008–102039`);  
- каждый шард покрывает диапазон значений хэша;  
- в `JOIN` с `pg_dist_node` мы видим, что шарды равномерно разложены по трём воркерам.

## Как работает распределение данных на твоём стенде

### Создание распределённой таблицы

Ты сделал:

```sql
CREATE TABLE dist_test (
    id   bigint,
    note text
);

SELECT create_distributed_table('dist_test', 'id');
```

Citus:

- зарегистрировал `dist_test` в `pg_dist_partition` как распределённую по хэшу по колонке `id`; 
- создал 32 шарда для этой таблицы и записал их в `pg_dist_shard`;
- разложил шарды по воркерам через `pg_dist_placement` и `pg_dist_node` (у тебя — равномерно по `citus_worker_1/2/3`).

### Вставка данных

При `INSERT INTO dist_test ... generate_series(1, 1000)`:

- для каждой строки считается хэш от `id`;  
- по таблице `pg_dist_shard` ищется шард, чей диапазон содержит это хэш‑значение; 
- по `pg_dist_placement` находится воркер, который этот шард хранит;  
- строка физически вставляется в таблицу `dist_test_<shardid>` на соответствующем воркере.

## Как выполняется запрос на твоём кластере

Возьмём запрос:

```sql
SELECT count(*) FROM dist_test;
```

Запускаешь ты его **на координаторе** (через pgAdmin или `psql`):

1. Координатор видит, что `dist_test` — распределённая таблица (смотрит в `pg_dist_partition`). 
2. Строит план: для каждого шарда `dist_test_<shardid>` на его воркере сделать `SELECT count(*)`. 
3. Рассылает под‑запросы на воркеры в несколько соединений.  
4. Собирает частичные результаты и суммирует. 

Для фильтра `WHERE id = 123` он:

- берёт значение `123`, считает хэш;  
- по диапазонам в `pg_dist_shard` находит один конкретный шард;  
- шлёт запрос только на тот воркер, который хранит этот шард. 

То есть ты всегда работаешь **только с координатором**, а он уже сам решает, какие воркеры задействовать.

## Почему ты «не видишь» шарды на воркере

Когда на воркере:

```sql
SELECT relname FROM pg_class WHERE relname LIKE 'dist_test_%';
```

и `(0 rows)`, это связано с тем, что:

- физические таблицы‑шарды могут быть в другой схеме и/или скрыты от прямого просмотра; 
- нормальный путь — смотреть шарды с координатора через `citus_shards` и `pg_dist_*`. 

Например, на координаторе:

```sql
SELECT logicalrelid, shardid, shard_name
FROM citus_shards
WHERE logicalrelid = 'dist_test'::regclass;
```

даст имена физических таблиц `dist_test_1020xx`, и зная `shardid` + ноду из `pg_dist_node`, ты уже можешь целенаправленно зайти на конкретного воркера.

## Как это применять в реальных проектах

- В бэкенде (FastAPI/Django) ты коннектишься к координатору как к обычному Postgres, драйвер/ORM даже не знает про Citus.
- Ключевые проектные решения:
  - выбор ключа распределения (часто user_id / tenant_id / order_id); 
  - какие таблицы делать распределёнными, какие — reference (мелкие справочники, реплицируются на все воркеры); 
  - как писать запросы так, чтобы они «локализовались» в один шард/воркер, когда возможно.
