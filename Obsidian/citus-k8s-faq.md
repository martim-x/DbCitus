## FAQ: Kubernetes + Citus


### 3 ноды и 5 подов — это 15 подов?

Нет. _**Нода**_ — машина, _**под**_ — экземпляр приложения. В StatefulSet задаёшь 5 подов, Kubernetes раскладывает их по 3 нодам. Количество подов не умножается на количество нод.


### В чём толк нескольких нод, если можно один контейнер Postgres?

Один контейнер = один процесс БД, один хост, один предел масштаба.

С Kubernetes + Citus:

- несколько нод → больше CPU/RAM/диска
- несколько воркеров → горизонтальное шардирование таблиц
- self-healing: упал pod/нода — Kubernetes пересоздаёт pod

Толк в масштабе и отказоустойчивости, чего один контейнер не даёт.


### `replicas: 5` в StatefulSet — это 5 копий БД (репликация)?

Нет. `replicas: 5` = запусти 5 **разных** worker-pod'ов (5 узлов Citus). Данные между ними по умолчанию не копируются. Репликация — это `citus.shard_replication_factor`, который задаёт, сколько копий **каждого шарда** хранится на разных воркерах.


### Если шарды не реплицировать, при падении ноды теряются данные?

Да. При `citus.shard_replication_factor = 1` (по умолчанию) каждый шард в одной копии — потеря воркера означает потерю его шардов. При факторе `2` каждый шард хранится на двух разных воркерах: потеря одной ноды не уничтожает данные.

> Без репликации шардов толк от нескольких нод — только в масштабе, не в отказоустойчивости.


### Шардирование и репликацию можно включить буквально двумя командами?

Ты задаёшь политику:

```sql
ALTER SYSTEM SET citus.shard_count = 32;
ALTER SYSTEM SET citus.shard_replication_factor = 2;
SELECT pg_reload_conf();
```

Чтобы она заработала, нужно ещё:

1. Зарегистрировать воркеров: `citus_add_node(...)`
2. Создать distributed-таблицу: `create_distributed_table(...)`

После этого новые distributed-таблицы автоматически шардируются и реплицируются по заданным параметрам.


### Нужно ли заходить на каждый воркер и настраивать шардирование там?

Нет. Всё делается **одним SQL-скриптом на координаторе**. Координатор хранит метаданные (`pg_dist_*`) и сам управляет размещением шардов по воркерам. На сами воркеры руками заходить не нужно.


### Почему при Stop/Reset в Docker Desktop всё слетает?

Docker Desktop воспринимает свой Kubernetes-кластер как временную сущность. При Stop/Reset удаляются control-plane/worker-контейнеры и связанные volumes. Если не используются persistent volumes вне кластера, данные БД лежат в ephemeral-дисках pod'ов и удаляются вместе с ними.

> Нормально для учебки. В проде — отдельное хранилище через PV/PVC.


### Почему имена воркеров такие длинные?

`citus-worker-0.citus-worker-headless.citus.svc.cluster.local` расшифровывается так:

- `citus-worker-0` — pod
- `citus-worker-headless` — headless Service
- `citus` — namespace
- `svc.cluster.local` — домен кластера

Headless Service (`clusterIP: None`) даёт DNS A-запись для **каждого пода отдельно**. Координатору нужны именно пер-pod-имена, чтобы регистрировать каждый воркер через `citus_add_node`.


### Как посмотреть, на какой ноде какой pod крутится?

```bash
kubectl -n citus get pods -o wide
```

В колонке `NODE` видно, на какой Kubernetes-ноде находится каждый pod.


### Как проверить, что шарды разложились по воркерам и репликация работает?

```sql
SELECT
    s.shardid,
    n.nodename,
    n.nodeport
FROM pg_dist_shard s
JOIN pg_dist_placement p ON s.shardid = p.shardid
JOIN pg_dist_node n      ON p.groupid = n.groupid
WHERE s.logicalrelid = 'events'::regclass
ORDER BY s.shardid, n.nodename;
```

Если `citus.shard_replication_factor = 2`, у каждого `shardid` в выводе две строки с разными `nodename` — это две копии шарда на разных воркерах.
